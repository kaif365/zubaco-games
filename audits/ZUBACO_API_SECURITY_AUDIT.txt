The API layer is the single most exposed surface of the Zubaco platform. With 
real money at stake, every endpoint must be fortified against abuse. Currently,
the platform has FUNDAMENTAL gaps in API security: no input sanitization layer,
client-trusted score submissions, race conditions in financial endpoints, and 
missing authorization checks on several sensitive operations.

│ API1:2023 - BROKEN OBJECT LEVEL AUTHORIZATION (BOLA)                            │
│ STATUS: ██████████ CRITICAL FAILURE                                             │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ FINDINGS:                                                                       │
│                                                                                 │
│ 1. GAME SESSION ACCESS (CRITICAL)                                               │
│    Endpoint: GET /game-session/:sessionId/state                                 │
│    Issue: No ownership check. Any authenticated user can query ANY session.     │
│    Code: game-session.controller.ts                                             │
│    ```                                                                          │
│    @Get(':sessionId/state')                                                     │
│    async getState(@Param('sessionId') sessionId: string) {                      │
│      // No userId check - returns any session's state                           │
│    }                                                                            │
│    ```                                                                          │
│    Impact: Attacker can enumerate session IDs to see other players' game        │
│    configs, server seeds, and scores during active tournaments.                 │
│                                                                                 │
│ 2. NOTIFICATION ACCESS (HIGH)                                                   │
│    Endpoint: POST /notifications/:id/read                                       │
│    Issue: No check that notification belongs to requesting user.                │
│    Impact: Can mark other users' notifications as read.                         │
│                                                                                 │
│ 3. BANK DETAIL DELETION (HIGH)                                                  │
│    Endpoint: DELETE /wallet/bank-details/:id                                    │
│    Issue: Must verify bank detail belongs to requesting user.                   │
│    If only checking ID existence, attacker can delete other users' bank info.   │
│                                                                                 │
│ 4. CHALLENGE ACCESS (MEDIUM)                                                    │
│    Endpoint: POST /social/challenges/:id/submit                                 │
│    Issue: Must verify user is challenger or opponent.                            │
│    Could allow third party to submit scores for others' challenges.             │
│                                                                                 │
│ 5. ANTI-CHEAT ADMIN ENDPOINTS (HIGH)                                            │
│    Endpoints: /anti-cheat/flags, /anti-cheat/users/:id/ban                      │
│    Issue: These are admin-only operations exposed in the platform backend.      │
│    Must use @UseGuards(AdminGuard) - currently only uses @UseGuards(AuthGuard). │
│    ANY authenticated user may be able to ban other users or review flags.       │
│                                                                                 │
│ BOLA SCORE: 15/100 (5 endpoints vulnerable)                                    │
│                                                                                 │

┌─────────────────────────────────────────────────────────────────────────────────┐
│ API2:2023 - BROKEN AUTHENTICATION                                               │
│ STATUS: ██████████ HIGH RISK                                                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ FINDINGS:                                                                       │
│                                                                                 │
│ 1. OTP BRUTE FORCE (CRITICAL)                                                   │
│    Endpoint: POST /auth/otp/verify                                              │
│    Issue: 6-digit OTP with maxAttempts=5. But verification is per-record,       │
│    not per-IP. Attacker can re-call /otp/send to generate new OTP and get       │
│    5 more attempts. No cooldown between OTP sends.                              │
│    Attack: Send 100 OTPs in 60 seconds, each gets 5 attempts = 500 guesses     │
│    against 1,000,000 possible codes = non-trivial success probability.          │
│                                                                                 │
│ 2. NO GOOGLE ID TOKEN VERIFICATION (CRITICAL)                                   │
│    Endpoint: POST /auth/google                                                  │
│    Code in auth.controller.ts:                                                  │
│    ```                                                                          │
│    @Post('google')                                                              │
│    async googleLogin(@Body() body: { id: string; email: string; ... }) {        │
│      return this.authService.googleLogin(body);                                 │
│    }                                                                            │
│    ```                                                                          │
│    The controller accepts raw body with Google profile data.                    │
│    THERE IS NO google-auth-library verification of the ID token!               │
│    Attacker can forge: { id: "any", email: "victim@gmail.com" }                │
│    and immediately get access tokens for that account.                          │
│                                                                                 │
│ 3. NO APPLE ID TOKEN VERIFICATION (CRITICAL)                                    │
│    Endpoint: POST /auth/apple                                                  │
│    Same pattern as Google - accepts raw profile without JWT verification.       │
│    Apple Sign-In requires server-to-server JWT validation.                      │
│                                                                                 │
│ 4. JWT FALLBACK SECRETS IN SOURCE CODE (HIGH)                                   │
│    File: config/index.ts                                                        │
│    ```                                                                          │
│    accessSecret: process.env.JWT_ACCESS_SECRET || 'dev-access-secret-change-me' │
│    refreshSecret: process.env.JWT_REFRESH_SECRET || 'dev-refresh-secret-change-me'│
│    ```                                                                          │
│    If env vars are missing, ALL JWTs are signed with known secrets.             │
│    Attacker can forge valid tokens for any user.                                │
│                                                                                 │
│ 5. NO DEVICE FINGERPRINTING (MEDIUM)                                            │
│    Token refresh doesn't validate that request comes from same device.          │
│    Stolen refresh tokens can be used from any location/device.                  │
│                                                                                 │
│ AUTHENTICATION SCORE: 20/100                                                    │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

│    GET /admin/exports/users exports: id, email, phone, username                │

│ API4:2023 - UNRESTRICTED RESOURCE CONSUMPTION                                   │
│ STATUS: ██████████ MEDIUM RISK                                                  │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ FINDINGS:                                                                       │
│                                                                                 │
│ 1. GLOBAL RATE LIMIT ONLY (MEDIUM)                                              │
│    app.module.ts: ThrottlerModule.forRoot([{ ttl: 60000, limit: 100 }])         │
│    Single rate limit for ALL endpoints: 100 requests/60 seconds per IP.         │
│    Issues:                                                                      │
│    - /auth/otp/send should be 3/minute (SMS cost attack)                        │
│    - /wallet/webhook/razorpay should be unlimited (Razorpay's IP)              │
│    - /leaderboard/* can be 1000/min (read-heavy, cacheable)                    │
│    - /wallet/withdraw should be 5/hour (financial abuse)                        │
│                                                                                 │
│ 2. NO PAGINATION LIMIT ON EXPORTS (HIGH)                                        │
│    GET /admin/exports/users fetches ALL users without limit:                    │
│    ```                                                                          │
│    const users = await this.prisma.user.findMany({                              │
│      where: { deleted_at: null },                                               │
│      select: { ... }                                                            │
│    });                                                                          │
│    ```                                                                          │
│    With 100K+ users, this OOMs the server or times out.                        │
│                                                                                 │
│ 3. LEADERBOARD QUERIES UNBOUNDED (MEDIUM)                                       │
│    If `limit` parameter isn't validated, client can request limit=1000000.      │
│                                                                                 │
│ 4. NO REQUEST BODY SIZE LIMIT (MEDIUM)                                          │
│    main.ts doesn't set app.use(json({ limit: '1mb' })).                        │
│    Attacker can send 100MB JSON bodies to exhaust memory.                       │
│                                                                                 │
│ 5. OTP SMS FLOODING (HIGH)                                                      │
│    POST /auth/otp/send has no per-phone-number rate limit.                     │
│    Attacker can trigger 10,000 SMS to a victim's phone number.                 │
│    Financial impact: ₹0.25-0.50 per SMS × 10K = ₹2,500-5,000 loss.           │
│                                                                                 │
│ RESOURCE CONSUMPTION SCORE: 40/100                                              │

│                                                                                 │
│ 1. ANTI-CHEAT ADMIN FUNCTIONS IN PLAYER API (CRITICAL)                          │
│    Platform backend exposes admin operations:                                   │
│    - POST /anti-cheat/users/:userId/ban                                        │

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 5.1 DEPOSIT FLOW SECURITY                                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Flow: create-order → Razorpay checkout → verify-payment / webhook              │
│                                                                                 │
│ VULNERABILITIES:                                                                │
│                                                                                 │
│ 1. DOUBLE-CREDIT RACE CONDITION (CRITICAL)                                      │
│    Both /deposit/verify AND /webhook/razorpay can credit the wallet.           │
│    If both arrive within same millisecond:                                     │
│    - verify finds PENDING transaction → credits wallet                         │
│    - webhook finds PENDING transaction → credits wallet again                  │
│    User deposited ₹100, received ₹200.                                        │
│                                                                                 │
│    Fix needed: Use database advisory locks or unique constraint on             │
│    reference_id + status transition. Transaction.update with                   │
│    { where: { status: 'PENDING' } } has a race window.                        │
│                                                                                 │
│ 2. AMOUNT MISMATCH (HIGH)                                                       │
│    /deposit/verify only checks signature, not that the AMOUNT matches.         │
│    Attack:                                                                      │
│    - Create order for ₹100                                                     │
│    - Modify Razorpay checkout to pay ₹10                                       │
│    - Submit valid signature for ₹10 payment                                    │
│    - Server credits ₹100 (stored in PENDING transaction)                       │
│    Fix: Verify payment amount via Razorpay API before crediting.               │
│                                                                                 │
│ 3. WEBHOOK REPLAY (MEDIUM)                                                      │
│    No idempotency key/nonce on webhook handling.                               │
│    If attacker captures a webhook payload, they can replay it.                 │
│    HMAC signature would still be valid.                                        │
│    Fix: Store processed webhook IDs, reject duplicates.                        │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 5.2 WITHDRAWAL FLOW SECURITY                                                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Flow: requestWithdrawal → PENDING → admin approval → processWithdrawal         │
│                                                                                 │
│ VULNERABILITIES:                                                                │
│                                                                                 │
│ 1. CONCURRENT WITHDRAWAL RACE (CRITICAL)                                        │
│    wallet.service.ts:                                                          │
│    ```                                                                          │
│    const balance = Number(wallet.balance);                                     │
│    if (balance < amount) throw ...;                                            │
│    const newBalance = balance - amount;                                        │
│    await this.prisma.$transaction([...]);                                      │
│    ```                                                                          │
│    Between reading balance and updating, another request can read              │
│    same balance. Two ₹1000 withdrawals from ₹1000 balance both succeed.       │
│                                                                                 │
│    Fix: Use SELECT ... FOR UPDATE or Prisma's atomic decrement:                │
│    ```                                                                          │
│    prisma.wallet.update({                                                      │
│      where: { user_id, balance: { gte: amount } },                            │
│      data: { balance: { decrement: amount } }                                  │
│    })                                                                           │
│    ```                                                                          │
│                                                                                 │
│ 2. NO WITHDRAWAL CONFIRMATION (HIGH)                                            │
│    No OTP/email verification for withdrawal requests.                          │
│    If attacker steals user's JWT:                                              │
│    - Add their bank details                                                    │
│    - Set as primary                                                            │
│    - Request withdrawal of full balance                                        │
│    All without any second factor.                                              │
│                                                                                 │
│ 3. NO COOLING PERIOD (MEDIUM)                                                   │
│    After changing bank details, withdrawal should be blocked for 24-48 hours.  │
│    Currently no such protection.                                               │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 5.3 ENTRY FEE SECURITY                                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Flow: register for season → deductEntryFee                                     │
│                                                                                 │
