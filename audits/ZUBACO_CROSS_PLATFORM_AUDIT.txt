┌──────────────────────────────────────────────────────────────────────────────┐
│ ISSUE XPLAT-AUTH-01: Token Refresh Broken Across Entire Stack                │
├──────────────────────────────────────────────────────────────────────────────┤
│ Severity: CRITICAL                                                           │
│ Affected: Mobile App ↔ Platform Backend                                     │
│                                                                              │
│ Chain of failure:                                                            │
│ 1. Platform issues access tokens with 15-minute expiry                      │
│ 2. Mobile app stores both access + refresh tokens                           │
│ 3. Mobile app NEVER calls refresh endpoint                                  │
│ 4. After 15 minutes, all API calls return 401                               │
│ 5. Mobile has no 401 interceptor → user sees cryptic errors                │
│ 6. User must manually logout and re-login                                   │
│                                                                              │
│ The refresh_token endpoint EXISTS on the platform backend but has its       │
│ own bugs (O(N) bcrypt scan of ALL user refresh tokens — PLATFORM-AUTH-03). │
│ Even if the mobile app called it, it would be slow.                        │
│                                                                              │
│ IMPACT: App is UNUSABLE for sessions longer than 15 minutes.               │
│                                                                              │
│ FIX CHAIN:                                                                   │
│ 1. Platform: Fix refresh token to use hashed lookup, not bcrypt scan       │
│ 2. Mobile: Add 401 interceptor → call refresh → retry original request    │
│ 3. Mobile: Add token expiry prediction (decode JWT, refresh before expiry) │
│                                                                              │
│ Priority Score: 10/10                                                        │
└──────────────────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────────────────┐
│ ISSUE XPLAT-GAME-01: Score Trust Chain is Completely Broken                  │
├──────────────────────────────────────────────────────────────────────────────┤
│ Severity: CRITICAL                                                           │
│ Category: Anti-Cheat                                                         │
│ Affected: Mobile → Game Frontend → Game Backend → Platform Backend          │
│                                                                              │
│ The score flow:                                                              │
│ 1. Mobile opens WebView with game URL + JWT token                           │
│ 2. Game frontend runs game logic IN THE BROWSER (client-side)              │
│ 3. Game frontend calls game backend with score + time                       │
│ 4. Game backend stores score, calls platform's /anti-cheat/submit-score    │
│ 5. Platform records score for tournaments/leaderboards                      │
│                                                                              │
│ Problems in the chain:                                                       │
│ A. Game logic runs client-side → scores are trivially fakeable             │
│ B. Game backends accept client-submitted scores without validation          │
│    (GAME audits found this in ALL 20 games)                                │
│ C. Platform anti-cheat endpoints have NO role-based access control          │
│    (PLATFORM-ANTICHEAT-01) — any authenticated user can call admin         │
│    anti-cheat endpoints                                                      │
│ D. Tournament scores go through same unvalidated path                       │
│ E. Mobile WebView injects JWT into window.__ZUBACO__ — game frontend      │
│    (or XSS in game) can extract token for other API calls                  │
│                                                                              │
│ For a REAL MONEY tournament platform, this is an existential threat.       │
│ Anyone with basic JavaScript knowledge can:                                 │
│ 1. Open browser DevTools                                                     │
│ 2. POST fake scores to game backend                                         │
│ 3. Win tournaments                                                           │
│ 4. Withdraw real money                                                       │
│                                                                              │
│ FIX (requires architectural change):                                         │
│ 1. Move critical game logic to backend (server-authoritative)              │
│ 2. At minimum: replay validation, statistical anomaly detection            │
│ 3. Rate-limit score submissions per session                                 │
│ 4. Use game-session-scoped tokens (not user JWT) for game backends         │
│                                                                              │
│ Priority Score: 10/10 (existential risk for real-money platform)            │
└──────────────────────────────────────────────────────────────────────────────┘

4. CRITICAL PATH ANALYSIS: "Can a User Play a Tournament Today?"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Let's trace the complete user journey and identify where it breaks:

STEP 1: User downloads app, opens it
  ✓ App.tsx loads, navigation works
  ✓ Onboarding flow renders

STEP 2: User logs in with phone OTP
  ✓ LoginScreen sends OTP (if platform backend is running)
  ✓ OtpScreen verifies OTP
  ✓ Tokens stored in AsyncStorage
  ⚠ Tokens expire in 15 min, no refresh mechanism

STEP 3: User adds money to wallet
  ✗ WalletScreen shows HARDCODED data (₹150 balance)
  ✗ "Add Money" button does nothing
  ✗ No Razorpay SDK integration
  ✗ BLOCKED — cannot deposit money

STEP 4: User registers for tournament
  ⚠ Tournament registration endpoint exists on platform
  ⚠ Mobile has TournamentScreen (unknown completeness)
  ✗ Cannot pay entry fee (wallet broken)
  ✗ BLOCKED — cannot register

STEP 5: User plays a game
  ⚠ Mobile opens WebView with game URL
  ✗ Game backends not deployed (no ECS services in Terraform)
  ✗ Game frontend API calls would fail (no backend)
  ✗ BLOCKED — game cannot communicate with backend

STEP 6: Score submitted
  ✗ Even if game worked, scores are client-trusted
  ✗ No server-side validation
  ✗ Trivially cheatable

STEP 7: Tournament results
  ⚠ Platform has tournament scoring logic
  ✗ Based on fake-able scores

STEP 8: User wins, withdraws money
  ✗ No KYC UI in mobile app
  ✗ No withdraw flow in mobile app
  ✗ BLOCKED — cannot withdraw

VERDICT: A user CANNOT complete a single real-money tournament today.
The critical path is broken at steps 3, 5, and 8.

    frontend source code.

    exceeds db.t3.medium limit (~85).

10. CI/CD DEPLOYMENT — force-new-deployment doesn't update image tag.
    Pushes code but doesn't deploy it.
