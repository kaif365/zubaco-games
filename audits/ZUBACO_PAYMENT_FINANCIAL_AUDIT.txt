FINANCIAL SYSTEM SCORE: 28/100 (CRITICAL - MONEY LOSS GUARANTEED)

For a platform where real money (INR) flows in and out, the financial subsystem
is alarmingly incomplete. Critical race conditions can cause double-credits, 
entry fees are never deducted, withdrawals lack second-factor verification, and 
there is ZERO reconciliation infrastructure. In production, this platform WILL 
lose money within hours of launch.

Key Metrics:
  - Transaction Types Supported: 6 (Deposit, Withdrawal, Entry Fee, Prize, Referral, Refund)
  - Payment Gateway: Razorpay (integration complete but insecure)
  - KYC System: Implemented (Aadhaar/PAN/DL/Passport) but no verification API
  - Financial Race Conditions: 3 CRITICAL
  - Missing Financial Controls: 14
  - Regulatory Compliance: 0% (no TDS, no GST, no RBI compliance)

             SECTION 1: DEPOSIT FLOW ANALYSIS

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 1.1 CURRENT DEPOSIT FLOW                                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Step 1: Client → POST /wallet/deposit/create-order { amount: 500 }             │
│ Step 2: Server → Razorpay API → Creates order (500 × 100 paise)               │
│ Step 3: Server → Creates PENDING transaction in DB                             │
│ Step 4: Server → Returns { order_id, amount, currency, key_id }               │
│ Step 5: Client → Opens Razorpay Checkout with order_id                        │
│ Step 6: Client → Completes payment on Razorpay                                │
│ Step 7: Client → POST /wallet/deposit/verify { order_id, payment_id, sig }    │
│ Step 8: Server → Verifies HMAC signature                                      │
│ Step 9: Server → Credits wallet + marks transaction COMPLETED                  │
│                                                                                 │
│ PARALLEL PATH:                                                                  │
│ Step 6b: Razorpay → POST /wallet/webhook/razorpay { event: payment.captured } │
│ Step 7b: Server → Verifies webhook signature                                  │
│ Step 8b: Server → Credits wallet + marks transaction COMPLETED                 │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-DEP-01: DOUBLE CREDIT (Verify + Webhook Race)                         │
│ Severity: CRITICAL                                                              │
│ Category: Race Condition                                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Both /deposit/verify and /webhook/razorpay execute:                            │
│                                                                                 │
│   const transaction = await prisma.transaction.findFirst({                     │
│     where: { reference_id: orderId, status: 'PENDING' }                       │
│   });                                                                          │
│   // ... credits wallet ...                                                    │
│   await prisma.transaction.update({ data: { status: 'COMPLETED' } });         │
│                                                                                 │
│ Timeline of attack:                                                            │
│   T+0ms:  /verify reads transaction → status is PENDING ✓                    │
│   T+1ms:  /webhook reads transaction → status is STILL PENDING ✓             │
│   T+5ms:  /verify updates wallet (+₹500) and sets COMPLETED                  │
│   T+6ms:  /webhook updates wallet (+₹500) and sets COMPLETED (already was)    │
│                                                                                 │
│ Result: User deposited ₹500, received ₹1000 in wallet.                        │
│                                                                                 │
│ Real-world likelihood: HIGH. Razorpay webhooks arrive within 1-5 seconds       │
│ of payment, and client verify call happens immediately after checkout closes.  │
│                                                                                 │
│ FIX: Use Prisma's updateMany with status filter (atomic):                      │
│   const result = await prisma.transaction.updateMany({                         │
│     where: { reference_id: orderId, status: 'PENDING' },                      │
│     data: { status: 'COMPLETED' }                                             │
│   });                                                                          │
│   if (result.count === 0) return; // Already processed                        │
│   // Only credit wallet if updateMany affected 1 row                          │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-DEP-02: AMOUNT NOT VERIFIED FROM RAZORPAY                             │
│ Severity: CRITICAL                                                              │
│ Category: Payment Fraud                                                         │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ /deposit/verify ONLY checks signature authenticity. It does NOT verify that    │
│ the actual payment amount matches the order amount.                            │
│                                                                                 │
│ Attack scenario:                                                                │
│ 1. Create order for ₹10,000 (100000 paise)                                    │
│ 2. Intercept Razorpay checkout                                                │
│ 3. Modify amount to ₹1 (100 paise) using browser devtools                     │
│ 4. Complete ₹1 payment                                                        │
│ 5. Submit valid signature for ₹1 payment                                      │
│ 6. Server credits ₹10,000 (amount from PENDING transaction)                   │
│                                                                                 │
│ The signature is valid (signed by Razorpay for ₹1 payment).                   │
│ But server trusts the PENDING transaction amount, not actual payment.          │
│                                                                                 │
│ FIX: After signature verification, call Razorpay API to fetch payment:        │
│   const payment = await fetch(`razorpay.com/v1/payments/${paymentId}`);        │
│   if (payment.amount !== transaction.amount * 100) { reject; }                │
│                                                                                 │
│ NOTE: Actually, Razorpay's order system PREVENTS amount modification.         │
│ The order is created server-side with fixed amount. But this should still     │
│ be verified as defense-in-depth.                                              │
│                                                                                 │

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-WD-01: CONCURRENT WITHDRAWAL DOUBLE-SPEND                             │
│ Severity: CRITICAL                                                              │
│ Category: Race Condition                                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Code in wallet.service.ts:                                                     │
│   const wallet = await this.getWallet(userId);                                │
│   const balance = Number(wallet.balance);                                      │
│   if (balance < amount) throw new BadRequestException(...);                    │
│   const newBalance = balance - amount;                                         │
│   await this.prisma.$transaction([...update wallet, create transaction]);      │
│                                                                                 │
│ Race:                                                                           │
│   T+0: Request A reads balance = ₹1000                                        │
│   T+1: Request B reads balance = ₹1000                                        │
│   T+2: Request A: 1000 >= 1000 ✓, sets balance to ₹0                         │
│   T+3: Request B: 1000 >= 1000 ✓, sets balance to ₹0                         │
│                                                                                 │
│ Result: User withdrew ₹2000 from ₹1000 balance.                               │
│ The second $transaction overwrites the first because it uses                   │
│ a CALCULATED newBalance, not an atomic decrement.                              │
│                                                                                 │
│ Financial loss: Unlimited. Script can fire 100 concurrent requests.            │
│                                                                                 │
│ FIX: Use Prisma atomic operations:                                             │
│   await prisma.wallet.update({                                                 │
│     where: { user_id: userId, balance: { gte: amount } },                     │
│     data: { balance: { decrement: amount } }                                  │
│   });                                                                          │
│   // Throws if balance insufficient (atomic check-and-decrement)              │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-PRIZE-02: NO TDS DEDUCTION ON WINNINGS                                │
│ Severity: CRITICAL (Legal)                                                      │
│ Category: Tax Compliance                                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Indian Income Tax Act Section 115BBJ (w.e.f. April 2023):                      │
│                                                                                 │
│ "Any person responsible for paying to any person any income by way of          │
│ winnings from any online game shall deduct TDS @ 30% on NET WINNINGS           │
│ at the time of withdrawal or at end of financial year, whichever is earlier."  │
│                                                                                 │
│ Net Winnings = Total withdrawals + closing balance - Total deposits            │
│                                                                                 │
│ The platform has ZERO TDS implementation:                                      │
│ - No TDS calculation on withdrawal                                             │
│ - No annual TDS computation                                                    │
│ - No TDS certificate generation (Form 16C)                                     │
│ - No PAN-based TDS tracking                                                   │
│ - No TDS filing with government                                                │
│                                                                                 │
│ Legal consequence: Platform operator is personally liable for unpaid TDS.      │
│ Penalty: 100% of TDS amount + interest + prosecution for evasion.             │
│                                                                                 │

│ - On each withdrawal: calculate net winnings                                  │

│ - File 26QC with Income Tax Department                                         │
│ - Provide TDS certificate to users                                            │
│                                                                                 │

             SECTION 5: KYC SYSTEM ANALYSIS

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-KYC-01: NO DOCUMENT VERIFICATION API                                  │
│ Severity: HIGH                                                                  │
│ Category: Compliance                                                            │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ kyc.service.ts only STORES documents. No verification happens.                 │
│                                                                                 │
│ Current flow:                                                                   │
│ 1. User uploads document photo (stored as S3 URL)                             │
│ 2. Document goes to PENDING state                                              │
│ 3. Admin manually reviews (in admin panel)                                    │
│ 4. Admin approves/rejects                                                      │
│                                                                                 │
│ Missing:                                                                        │
│ - Aadhaar verification via UIDAI API or DigiLocker                            │
│ - PAN verification via NSDL/Income Tax API                                    │
│ - Liveness detection (prevent photo-of-photo)                                 │
│ - OCR extraction and cross-matching                                           │
│ - Name matching between document and account                                  │
│ - Age verification (must be 18+)                                              │
│                                                                                 │
│ For a real-money gaming platform, manual KYC review is:                       │
│ - Too slow (users wait days for withdrawal)                                   │
│ - Not scalable (1 admin for 100K users?)                                      │
│ - Prone to human error (fake documents approved)                              │
│                                                                                 │
│ Recommended APIs:                                                               │
│ - Digio (India): Aadhaar eKYC + PAN verification                              │
│ - HyperVerge: Document OCR + face match + liveness                            │
│ - Signzy: Full-stack digital KYC                                              │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-KYC-02: DOCUMENT STORED AS PLAIN S3 URL                               │
│ Severity: HIGH                                                                  │
│ Category: Data Security                                                         │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ KycDocument model:                                                              │
│   document_url: String  // S3 URL                                             │
│                                                                                 │
│ If this is a public S3 URL, anyone with the link can access the document.     │
│ Aadhaar/PAN/Passport photos are extremely sensitive PII.                      │
│                                                                                 │
│ Required:                                                                       │
│ - S3 bucket must be private (no public access)                                │
│ - Generate pre-signed URLs with 5-minute expiry for admin viewing             │
│ - Encrypt documents at rest (S3-SSE-KMS)                                      │
│ - Implement access logging for compliance                                     │
│ - Delete documents after KYC approval (data minimization)                     │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-KYC-03: NO AGE VERIFICATION                                           │
│ Severity: CRITICAL (Legal)                                                      │
│ Category: Regulatory                                                            │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Online real-money gaming in India requires users to be 18+.                    │
│ Multiple states (Andhra Pradesh, Telangana, Tamil Nadu) have additional        │
│ restrictions on online gaming.                                                 │
│                                                                                 │
│ Current implementation: ZERO age verification.                                  │
│ - No DOB field in User model                                                  │
│ - No age check during registration                                            │
│ - No DOB extraction from KYC documents                                        │
│ - No self-declaration of age                                                  │
│                                                                                 │
│ A 14-year-old can register, deposit money, and play in tournaments.            │
│ This is a criminal offense under state gaming laws.                            │
│                                                                                 │
│ MINIMUM required: DOB field + age >= 18 check at registration.                 │
│ RECOMMENDED: Extract DOB from Aadhaar/PAN and verify automatically.           │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

             SECTION 6: RECONCILIATION & ACCOUNTING

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-RECON-01: ZERO RECONCILIATION SYSTEM                                  │
│ Severity: HIGH                                                                  │
│ Category: Financial Operations                                                  │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ A financial platform MUST reconcile:                                            │
│                                                                                 │
│ 1. Razorpay settlements vs internal deposits                                  │
│    "Did we credit every payment that Razorpay confirmed?"                     │
│    "Did we credit any payment that Razorpay didn't confirm?"                  │
│                                                                                 │
│ 2. Wallet balances vs transaction history                                      │
│    SUM(deposits) - SUM(withdrawals) - SUM(entry_fees) + SUM(prizes)            │
│    + SUM(referral_bonuses) = current_balance                                   │
│    Must be TRUE for every user at all times.                                  │
│                                                                                 │
│ 3. Total platform liability vs total deposits received                         │
│    SUM(all_wallet_balances) <= SUM(razorpay_settlements) + prize_fund          │
│                                                                                 │
│ Currently: NONE of these checks exist.                                         │
│ - No daily reconciliation job                                                  │
│ - No balance integrity check                                                  │
│ - No Razorpay settlement matching                                             │
│ - No financial dashboard showing platform P&L                                 │
│                                                                                 │
│ When a double-credit bug fires (Section 1), nobody will know until            │
│ users withdraw more than was deposited.                                       │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-RECON-02: NO REFUND MECHANISM                                         │
│ Severity: MEDIUM                                                                │
│ Category: Operations                                                            │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ TransactionType includes REFUND, but no refund endpoint or service exists.    │
│                                                                                 │
│ Scenarios requiring refunds:                                                    │
│ - Tournament cancelled after entry fee deducted                               │
│ - Server error during game (score lost)                                       │
│ - Duplicate deposit credited (needs partial refund)                           │
│ - User dispute resolution                                                     │
│ - Chargebacks from Razorpay                                                   │
│                                                                                 │
│ Currently: Admin would need direct database access to issue refunds.           │
│ No audit trail, no approval workflow, no user notification.                    │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-RECON-03: BALANCE_AFTER COLUMN UNRELIABLE                             │
│ Severity: MEDIUM                                                                │
│ Category: Data Integrity                                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Every Transaction record stores balance_after.                                 │
│ But this is calculated at creation time:                                       │
│   balance_after: currentBalance + amountInr (for deposits)                    │
│                                                                                 │
│ If transactions process out of order (race conditions), balance_after          │
│ values become inconsistent. Example:                                           │
│                                                                                 │
│ Transaction 1: deposit ₹100, balance_after: ₹600 (correct)                    │
│ Transaction 2: deposit ₹200, balance_after: ₹700 (wrong - should be ₹800)    │
│ (Both read balance as ₹500 simultaneously)                                    │
│                                                                                 │
│ This makes audit trails unreliable.                                            │
│                                                                                 │
│ FIX: Calculate balance_after from previous transaction's balance_after,        │
│ not from wallet.balance at time of creation.                                  │
│ OR: Remove balance_after and compute it on-the-fly from running total.        │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ ISSUE FIN-BANK-01: BANK ACCOUNT NUMBERS STORED UNENCRYPTED                      │
│ Severity: HIGH                                                                  │
│ Category: Data Security                                                         │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ BankDetail model:                                                               │
│   account_number: String?                                                      │
│   ifsc_code: String?                                                           │
│   upi_id: String?                                                              │
│                                                                                 │
│ All stored as plaintext in PostgreSQL.                                          │
│ Database breach = all users' bank account numbers exposed.                     │
│                                                                                 │
│ Required:                                                                       │
│ - Encrypt account_number at application level (AES-256-GCM)                   │
│ - Store only last 4 digits for display purposes                               │
│ - IFSC can remain plaintext (public information)                              │
│ - UPI IDs are semi-public but should still be protected                       │
│ - Implement column-level encryption or use AWS KMS envelope encryption        │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

├─────────────────────────────────────────────────────────────────────────────────┤

│                                                                                 │

│ Industry standard:                                                              │
│ - Penny drop (₹1 credit + verify) using Razorpay Fund Account Validation     │
│ - Name matching between KYC name and bank account holder name                 │
│ - Account must be in user's own name (regulatory requirement)                 │
