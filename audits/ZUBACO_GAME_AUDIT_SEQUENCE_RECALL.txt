ZUBACO GAME AUDIT — GAME #16: SEQUENCE RECALL (Simon Says / Memory Sequence)
================================================================================
Audit Date: 2025-05-23
Auditor: AAA Game QA / Security / Performance / UX Engineering Review
Game Type: Memory sequence — tiles flash in a pattern, player replays from
          memory. Sequence length increases each round. Timer-survival mode
          with time bonus for completing without errors.
Stack: NestJS 11 + Prisma 7 + Restate (durable execution) + Redis + BullMQ +
       Zod + SNS (backend), React 19 + Vite 6 + Framer Motion + TanStack Query
       + react-i18next + Web Audio (frontend)

1. EXECUTIVE SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Sequence Recall is the FLAGSHIP game of the ZUBACO platform. It is
DRAMATICALLY more sophisticated than every other game — implementing a full
Restate-based durable execution workflow, server-authoritative per-move
validation, a 4-detector anti-cheat system, proper audio with multiple sound
packs, internationalization, session recovery on page refresh, tutorial/demo
mode, configurable wrong-move handling strategies, time bonus system, and
AES-256-GCM payload encryption.

The architecture is production-grade. The Restate Virtual Object ensures
exactly-once semantics even under retries. Each tile tap is validated server-
side in real-time. The anti-cheat system detects: (1) click-too-fast bots,
(2) uniform timing (low coefficient of variation), (3) impossible round
solve times, and (4) invalid end state (move count mismatch). Cheat flags
are published to SNS for cross-service processing.

The frontend has proper phase management (8 phases), audio feedback per
tile, wrong-tile visual cue, sequence dots progress indicator, session timer
with warning/critical states, and a robust session persistence + recovery
system.

Main issues: .env committed, serverSeed unnecessarily sent to client, HMAC
fallback secret, and the GameShell component is overly complex (~800+ lines).

4. PERFORMANCE ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BACKEND Performance:

1. Restate Virtual Object: Serializes requests per session key — eliminates
   race conditions but limits throughput to sequential processing per player.
   This is INTENTIONAL and correct for game state consistency.

2. SHA-256 Hash Chain: O(n) for generating sequence of length n. Cached in
   Restate state (cachedSequence) so computed only ONCE per session. Excellent.

3. Anti-Cheat: Runs asynchronously after finalization. Does NOT block the
   response to the player. Zero latency impact on gameplay.

4. Database: One write at session creation, one at finalization. During
   gameplay, ALL state lives in Restate's in-memory store. Minimal DB load.

5. SNS Publishing: Async, fire-and-forget. No gameplay latency impact.

FRONTEND Performance:

1. Sequence Playback: async/await with setTimeout delays. Clean, cancelable.
2. Framer Motion: Leaf tile animations are simple scale/opacity. Lightweight.
3. TanStack Query: Proper cache management, no unnecessary re-fetches.
4. Audio: Preloaded sounds (preload: true in registry). No loading delay.
5. Re-renders: The 30+ useState calls could cause frequent re-renders.
   MITIGATED: Most are booleans/numbers that only change on user interaction.

VERDICT: Performance is EXCELLENT. Restate + cached sequence + async anti-cheat
means near-zero latency per move validation. The only concern is the large
component tree re-render on each state change.

9. SECURITY RISK ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

THREAT MODEL:

Attack Surface 1: Anti-Cheat
- serverSeed sent → SHA-256 chain reproducible → all rounds known in advance
- MITIGATED: Server validates each move in real-time (can't skip tiles)
- MITIGATED: 4-detector anti-cheat system catches bots
- MITIGATED: Even with sequence knowledge, player must still submit tiles
  one-by-one at human speed (click-too-fast detection)
- Anti-cheat score: 75/100 (BEST IN PLATFORM!)

Attack Surface 2: Secrets
- .env committed (JWT + ENCRYPTION_KEY + DB credentials)
- HMAC fallback ('dev-hmac-secret')
- MITIGATED: env.validation.ts enforces required vars in production
- Severity: CRITICAL

Attack Surface 3: Authentication
- Dev session auto-refresh on 401 (TODO(temp) workaround)
- Token in localStorage (XSS risk if site is compromised)
- Severity: HIGH (must remove dev workaround)

Attack Surface 4: DoS
- Restate serializes per-session (natural throttle)
- No global rate limiter
- Severity: MEDIUM

Attack Surface 5: Data Integrity
- Restate durable execution ensures consistency
- P2002 handling prevents duplicate sessions
- Fire-and-forget anti-cheat could miss flags
- Severity: LOW

OVERALL SECURITY POSTURE:
BEST IN PLATFORM. Server-authoritative per-move validation, durable execution,
4-detector anti-cheat with SNS publishing, encrypted payloads, HMAC-guarded
admin. Only weakness: seed exposure and standard .env issues.
