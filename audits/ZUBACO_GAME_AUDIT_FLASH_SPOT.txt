================================================================================

Game Type: Reaction-time observation game — tap cells that are actively changing
          (color, shape, size, opacity) within a timed window

       React 19 + Vite 6 + Framer Motion + TailwindCSS (frontend)

1. EXECUTIVE SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flash Spot is a well-architected reaction game with server-authoritative scoring,
deterministic seeded change schedules, and solid anti-cheat foundations. The
backend is one of the more complete in the lineup — proper AES-GCM encryption,
atomic CAS pattern, Restate durable execution, idempotency keys, and circuit
breakers.

However, it has a UNIQUE CRITICAL BUG that no other game has: the encryption
env var name is MISMATCHED between config (.env uses VITE_PAYLOAD_AES_SECRET)
and code (httpClient.ts reads VITE_ENCRYPTION_KEY). This means encryption is
SILENTLY NEVER APPLIED — all payloads sent in plaintext despite the developer's
clear intent to encrypt them.

Additionally, using Date.now() for reaction timing in a game that MEASURES
reaction time is fundamentally wrong — performance.now() is required. And the
requestAnimationFrame loop sets React state EVERY FRAME (60fps), creating a
constant 60 re-renders/second that's 6× worse than the timer issue in other games.

TOTAL ISSUES: 60 (4 Critical, 13 High, 20 Medium, 23 Low)

RELEASE READINESS SCORE: 56/100
(Strong backend architecture undermined by env var mismatch + timing inaccuracy)

4. PERFORMANCE ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Performance Bottlenecks (sorted by impact):

1. requestAnimationFrame Loop = 60 Re-renders/Second (WORST ISSUE)
   - setActiveChanges(new Set(...)) called every frame
   - New Set object triggers React reconciliation every 16ms
   - GameGrid (16 cells) + header + controls re-render each time
   - This is 6× WORSE than the 100ms timer in other games
   - Fix: Only setState when active set content actually changes

2. changes.find() Called Per Cell Per Frame (O(n² per frame))
   - 16 cells × 12 changes × 60fps = 11,520 comparisons/second
   - Fix: Pre-compute lookup Map once per state change

3. Timer setInterval Drifts vs rAF Loop
   - Two independent timing systems running simultaneously
   - Under load: visible desync
   - Fix: Consolidate to single rAF-based timing

4. Date.now() Jitter (1-15ms resolution)
   - Not a performance issue but an ACCURACY issue
   - In a reaction game, 15ms error is significant
   - Fix: performance.now() (sub-millisecond resolution)

5. framer-motion in Main Chunk (~150KB)
   - Increases initial load time
   - Fix: Dynamic import or manual chunk

ESTIMATED PERFORMANCE PROFILE:
- Idle (pre-game): 0ms/frame (fine)
- During gameplay: ~12ms render work per 16ms frame (75% budget consumed!)
- On budget phone (2× slower): ~24ms per frame = CONSTANT JANK
- With F-06 fix: Renders only on state change (~2-3x per second) = smooth

UX PROBLEMS:

1. Date.now() Timing Inaccuracy (CRITICAL FOR THIS GENRE)
   - This is a REACTION GAME — timing IS the core mechanic
   - Using an inaccurate clock defeats the game's purpose
   - Players will feel "I tapped it!" but get incorrect result
   - FRUSTRATION LEVEL: VERY HIGH

2. No Haptic/Audio Feedback on Tap
   - Reaction games NEED instant feedback
   - Visual-only confirmation adds latency to perception
   - Players can't tell if tap registered without looking
   - FRUSTRATION LEVEL: HIGH

3. No Double-Tap Protection
   - Rapid tapping exploits score
   - Breaks fairness of leaderboard
   - Honest players at disadvantage
   - FRUSTRATION LEVEL: HIGH (for honest players)

4. Replay Bug (State Not Reset)
   - Playing again carries over old state
   - Score contamination
   - FRUSTRATION LEVEL: HIGH

5. No Pause on Background
   - Phone notification → game ruined
   - Tab switch → timer expired
   - FRUSTRATION LEVEL: MEDIUM

6. No Sound Effects
   - useAudio hook exists but disconnected
   - Silent reaction game feels incomplete
   - FRUSTRATION LEVEL: MEDIUM

9. SECURITY RISK ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

THREAT MODEL:

Attack Surface 1: Encryption (COMPLETELY BROKEN)
- VITE_ENCRYPTION_KEY ≠ VITE_PAYLOAD_AES_SECRET (env var mismatch)
- Encryption SILENTLY NEVER APPLIED — all traffic plaintext
- Even if fixed: key exposed in JS bundle (VITE_ prefix)
- Severity: CRITICAL (unique to Flash Spot)

Attack Surface 2: Secret Management
- 3 different fallback JWT secrets across HTTP/WS/config
- HMAC fallback 'dev-hmac-secret'
- .env committed pattern (identical to .env.example)
- Severity: CRITICAL

Attack Surface 3: Anti-Cheat
- Seed returned to client enables pre-computation
- Client timestamps fully trusted for tap validation
- Client isCorrect stored as-is (analytics poisoning)
- Double-tap exploit (no deduplication)
- Severity: HIGH

Attack Surface 4: Authentication
- Token in sessionStorage (XSS vulnerable)
- Socket uses localStorage / HTTP uses sessionStorage (mismatch)
- Source maps shipped to production (aids exploitation)
- Severity: HIGH

Attack Surface 5: Infrastructure
- /metrics publicly exposed
- In-memory rate limiter (bypassed + leaks)
- BullMQ connection missing Redis auth
- timingSafeEqual crash on length mismatch
- Severity: MEDIUM

OVERALL SECURITY POSTURE:
Server-side change schedule regeneration + scoring is excellent architecture.
But the encryption being SILENTLY BROKEN means all that work is undermined.
The env var name mismatch is a subtle, dangerous bug that tests would miss
(encryption "works" — it just uses empty key, which may pass without error).
