REAL-TIME SCORE: 5/100 (CRITICAL - ZERO REAL-TIME CAPABILITY)

A gaming platform REQUIRES real-time communication for competitive gameplay,
live leaderboards, tournament notifications, and player-vs-player interaction.
Zubaco has ZERO real-time infrastructure. No WebSocket server, no Socket.io,
no Server-Sent Events, no push notification backend. The platform operates
entirely on REST polling, which is inadequate for a competitive gaming experience.

REAL-TIME FEATURES NEEDED (INDUSTRY STANDARD):
  ❌ Live leaderboard updates during tournaments
  ❌ Real-time tournament stage transitions
  ❌ Live challenge notifications
  ❌ In-game multiplayer communication
  ❌ Push notifications for tournament events
  ❌ Real-time wallet balance updates
  ❌ Live anti-cheat alerts to admins
  ❌ Player online/offline status

             SECTION 3: RECOMMENDED ARCHITECTURE

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 3.1 IMPLEMENTATION OPTIONS                                                      │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Option A: NestJS WebSocket Gateway (Socket.io)                                 │
│   Pros: Native NestJS support, same codebase, guards/pipes work              │
│   Cons: Sticky sessions needed with multiple ECS tasks                        │
│   Cost: Redis adapter for multi-instance ($0 - uses existing Redis)           │
│   Effort: 2-3 weeks                                                           │
│                                                                                 │
│ Option B: AWS API Gateway WebSocket                                             │
│   Pros: Serverless, auto-scaling, managed                                     │
│   Cons: Connection limit (500 concurrent default), Lambda cold starts        │
│   Cost: $1/million messages + connection minutes                             │
│   Effort: 3-4 weeks                                                           │
│                                                                                 │
│ Option C: Third-party (Pusher, Ably, Firebase Realtime)                        │
│   Pros: Zero infrastructure, SDK for mobile                                   │
│   Cons: Vendor lock-in, per-message pricing (expensive at scale)             │
│   Cost: $49-499/month depending on connections                               │
│   Effort: 1 week                                                              │
│                                                                                 │
│ RECOMMENDED: Option A (NestJS WebSocket Gateway)                               │
│   - Already familiar tech stack                                               │
│   - Uses existing Redis for pub/sub adapter                                   │
│   - Full control, no vendor limits                                            │
│   - JWT auth can be shared                                                    │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐

┌─────────────────────────────────────────────────────────────────────────────────┐
│ Current ALB supports WebSocket natively (HTTP upgrade).                         │
│ However, multi-task ECS requires:                                              │
│                                                                                 │
│ 1. STICKY SESSIONS on ALB target group (connection affinity)                  │
│    OR                                                                          │
│ 2. REDIS ADAPTER for Socket.io (broadcast across instances)                   │
│                                                                                 │
│ Current Terraform ALB config:                                                  │
│   - stickiness not configured                                                 │
│   - WebSocket idle timeout: default 60s (should be 3600s for gaming)         │
│                                                                                 │
│ NEEDED CHANGES:                                                                 │
│   - ALB idle_timeout: 3600 (for long-lived WS connections)                   │
│   - Target group stickiness OR Redis adapter                                  │
│   - Health check path for WebSocket service                                   │
│   - Separate target group for WebSocket traffic                              │
│                                                                                 │
