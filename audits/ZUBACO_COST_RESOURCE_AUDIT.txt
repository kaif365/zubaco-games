━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

COST PLANNING SCORE: 35/100 (MEDIUM RISK - UNDERSIZED + NO BUDGET)

The infrastructure is significantly UNDERSIZED for the number of services being
deployed. A single db.t3.medium (85 max connections) serving 22 microservices is
a bottleneck waiting to happen. No cost estimation, no budget alerts, and no 
scaling plan exist. The platform will either crash under load or cost 3-5x more
than expected when auto-scaling kicks in without limits.

ESTIMATED MONTHLY COSTS (at current config):
  Minimal load (development):     $400-600/month
  Low load (100 DAU):             $800-1,200/month
  Medium load (1,000 DAU):        $2,000-4,000/month
  High load (10,000 DAU):         $8,000-15,000/month
  Scale (100,000 DAU):            $40,000-80,000/month (requires architecture changes)

│ TOTAL (minimum)     │       │        │ 23  │ 55  │ $431-1520                   │
│                                                                                 │
│ ISSUES:                                                                         │
│                                                                                 │
│ 1. 22 ALWAYS-ON SERVICES (HIGH COST)                                            │
│    Even at minimum (1 task each), that's 22 ECS tasks running 24/7.           │
│    Each game backend serves maybe 1-10 requests/hour at launch.               │
│    Cost: $360/month for 20 game backends that are 99% idle.                   │
│                                                                                 │
│ 2. NO AUTO-SCALING TO ZERO                                                      │
│    Fargate minimum is 1 task. Can't scale to 0 when idle.                     │
│    Alternative: Lambda + API Gateway for game backends (pay-per-request).     │
│                                                                                 │
│ 3. GAME BACKENDS SHOULD BE CONSOLIDATED                                         │
│    20 identical NestJS apps with 90% same code.                               │
│    Could be 1 service with game_type routing.                                 │
│    Reduces: 20 services → 1 service (20× cost reduction for games)            │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 1.2 DATABASE (RDS PostgreSQL)                                                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ Instance: db.t3.medium                                                         │
│ vCPU: 2                                                                        │
│ Memory: 4 GB                                                                   │
│ Max connections: 85                                                            │
│ Storage: 50GB (auto-scale to 200GB)                                           │
│ Multi-AZ: Yes (production)                                                     │
│ Cost: ~$140/month (Multi-AZ)                                                  │
│                                                                                 │
│ CRITICAL ISSUE: CONNECTION EXHAUSTION                                           │
│                                                                                 │
│ 22 services × 1 task × Prisma default pool (5) = 110 connections needed       │
│ db.t3.medium max connections = 85                                              │
│ IMMEDIATE FAILURE at minimum load.                                             │
│                                                                                 │
│ With auto-scaling:                                                             │
│ 22 services × 3 tasks × Prisma pool (5) = 330 connections needed             │
│ Even db.r6g.large (1365 max connections) would struggle.                      │
│                                                                                 │
│ SOLUTIONS:                                                                      │
│   Option A: RDS Proxy ($140/month) - connection pooling at AWS level          │
│   Option B: Consolidate services (fewer connection pools)                     │
│   Option C: Reduce Prisma pool size to 2 per service                          │
│   Option D: Upgrade to db.r6g.large ($400/month Multi-AZ)                     │
│                                                                                 │
│ RECOMMENDED: Consolidate game backends (20→1) + RDS Proxy                      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
