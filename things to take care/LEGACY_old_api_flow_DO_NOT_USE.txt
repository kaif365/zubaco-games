Game Flow Documentation
Overview
This document explains the complete game flow between the Shunya App, the Backend APIs, and the
Game Provider.
The key  is generated using the following format:
key = contestId::playerId
Example:
eae5816f-f3d0-461e-8030-c10ee810c170::104
Where:
contestId  = eae5816f-f3d0-461e-8030-c10ee810c170
playerId  = 104
1. Start Game
The game is initiated from the Shunya App.
The app calls the startGame  API to create a game session for the player.
API Request
curl --location 'http://localhost:8088/UserContest/eae5816f-f3d0-461e-803
--header 'Authorization: Bearer <token>' \
--header 'Content-Type: application/json' \
--data '{
  "providerId": "a0a08d9d-1055-4ef4-bc9c-a3605f3c8467",

2. Open Game in WebView
After receiving the response from the startGame  API:
1. The Shunya App opens the gameUrl  inside a WebView.
2. The app injects the following values into the game:
token
gameSessionId
key
These values are required by the Game Provider to authenticate and identify the active player session.
3. Submit Game Result
Once gameplay is completed, the Game Provider submits the result using the submitGameResult
API.
API Request
curl --location 'http://localhost:8088/UserContest/eae5816f-f3d0-461e-803
--header 'Content-Type: application/json' \
--data '{
  "gameSessionId": "58e6b3a0-f822-43bb-b97e-1a78377bd1f4",
  "outcome": "completed",
  "score": 100
}'

Important Notes
gameSessionId  is unique for every game session.
key  uniquely identifies the contest-player combination.
Authentication is handled using the Authorization: Bearer <token>  header.
The Game Provider should always use the same gameSessionId  received during game
initialization.
The submitGameResult  API should only be called after gameplay completion.
The injected token , gameSessionId , and key  should be securely handled inside the
WebView/game environment.
