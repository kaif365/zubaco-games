# Internationalization (i18n)

This project uses `i18next` + `react-i18next` with TypeScript locale bundles and locale selection from URL query params.

## Where translations live

- `src/locales/en/index.ts`
- `src/locales/hi/index.ts`

Both files should keep the same key structure.

## Locale resolution order

Locale is resolved in this order:

1. `?lang=en` or `?lang=hi`
2. `il_lang` cookie (persisted from a previous query selection)
3. default locale: `en`

## Runtime flow

1. `src/lib/i18n/provider.tsx` (client):
   - reads `?lang=...`
   - falls back to `il_lang` cookie, then `en`
   - persists resolved locale back to `il_lang` cookie
   - applies language via `i18n.changeLanguage(...)`
2. `src/app/layout.tsx` (server):
   - reads `il_lang` cookie via `next/headers`
   - sets `<html lang={locale}>` and passes `lng` to `I18nProvider`
3. `src/lib/i18n/server.ts`:
   - `getTranslation()` reads `il_lang` cookie (when request scope exists)
   - falls back to `en` otherwise (e.g. tests)

## Main APIs

### `src/lib/i18n/server.ts`

- `getTranslation()` is the only server i18n API.
- It creates a scoped i18next instance and returns `{ t, i18n }`.
- It resolves locale from `il_lang` cookie with `en` fallback.

## How to test locally

- English: `http://localhost:3000/?lang=en`
- Hindi: `http://localhost:3000/?lang=hi`

After opening a URL with `?lang=...`, subsequent navigations without query keep that locale via cookie.

## Adding a new key

1. Add key to `src/locales/en/index.ts` and `src/locales/hi/index.ts`.
2. Use `t("section.key")` in UI for string leaves.
3. For structured payloads (instructions/results), use `i18n.t(..., { returnObjects: true })` in the consuming component.
4. Run:

```bash
npm run build
npx vitest run tests/unit/translations.test.ts
```
