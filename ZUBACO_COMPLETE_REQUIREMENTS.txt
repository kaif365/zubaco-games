1. PLATFORM OVERVIEW
================================================================================

Concept: Mass-participation skill-based elimination gaming platform.
         Millions of players -> 5 online stages -> Seasonal winners.

Core Loop:
  - User downloads app -> plays free practice games (levels)
  - Joins seasonal tournament -> plays through 5 stages
  - Each stage: play 4 games, get scored, bottom 50% eliminated
  - Top performers win prizes/recognition

Two Modes:
  A) FREE PLAY - Practice games with progressive levels (always available)
  B) TOURNAMENT - Seasonal elimination competition (timed windows)

Revenue Model (to decide later):
  - Entry fees for tournaments
  - In-app purchases (hints, extra lives in free play)
  - Brand sponsorships
  - Ad-supported free tier

2. GAME ROSTER - FINAL 20 AI-PROOF GAMES
================================================================================

All cheatable games (MCQs, Sudoku, Spot the Difference, Path Optimisation,
Connections) have been REMOVED and replaced with AI-proof alternatives.

STAGE 1 - MEMORY & ATTENTION
------------------------------
Game 1: Sequence Recall (Simon Says)                    [BUILT]
  - Grid of cells flash in sequence -> player repeats from memory
  - Sequence grows +1 each round until failure
  - AI-proof: real-time flash, no static screenshot possible

Game 2: Memory Card Matching                            [BUILT]
  - Cards shown briefly -> flipped face down -> find matching pairs
  - AI-proof: card positions only visible for ~3 seconds

Game 3: Flash Spot (replaces Spot the Difference)       [TO BUILD]
  - ONE image displayed -> changes animate IN REAL-TIME
  - Tap the changing elements as they happen
  - AI-proof: no static screenshot, changes are animated/moving

Game 4: Object Placement Memory                         [TO BUILD]
  - Objects shown on grid for 5 seconds -> disappear
  - Player must place them back from memory
  - AI-proof: display phase too brief, AI never sees positions

STAGE 2 - SPEED & REACTION
----------------------------
Game 5: Sliding Puzzle                                  [BUILT]
  - Arrange tiles in correct order by sliding
  - AI-proof: even if AI solves it, user must physically swipe tiles

Game 6: Block Fill (Flow Free)                          [BUILT]
  - Connect colored dots without overlapping paths
  - AI-proof: must draw paths manually under time pressure

Game 7: Colour-Based Sorting                            [TO BUILD]
  - Sort colored balls into tubes (Ball Sort Puzzle)
  - AI-proof: requires physical tap/drag for each ball

Game 8: Rapid Category Sort (replaces MCQ Math)         [TO BUILD]
  - Items fall from top, swipe LEFT/RIGHT to sort into categories
  - Items appear every 1.5 seconds - no time to consult AI
  - AI-proof: continuous real-time input required

STAGE 3 - LOGIC & DUAL-TASK
-----------------------------
Game 9: Maze Navigation                                 [BUILT]
  - Navigate player through procedurally generated maze
  - AI-proof: real-time swipe navigation, AI cannot swipe for you

Game 10: Infinity Loop                                  [BUILT]
  - Rotate pipe segments to form complete loops
  - AI-proof: requires many individual taps/rotations

Game 11: Word Unscramble Race (replaces MCQ Logic)      [TO BUILD]
  - Scrambled letters shown -> tap in correct order to form word
  - New word every 6 seconds whether solved or not
  - AI-proof: physical input speed is the bottleneck

Game 12: True or False Blitz (replaces MCQ Mythology)   [TO BUILD]
  - Statements flash for 2 seconds each, auto-advance
  - Tap GREEN (true) or RED (false) before it moves on
  - 30 statements in 60 seconds
  - AI-proof: 2 seconds per question - no time to screenshot + ask AI

STAGE 4 - PERSISTENCE & STRATEGY
----------------------------------
Game 13: Arrows                                         [BUILT]
  - Clear arrows by matching direction patterns
  - AI-proof: requires rapid tapping in correct sequence

Game 14: Logic Reflector                                [BUILT]
  - Redirect beams using mirrors/reflectors to hit targets
  - AI-proof: trial-and-error physical placement

Game 15: Number Grid Sprint (replaces Sudoku)           [TO BUILD]
  - 6x6 grid where cells PULSE/BLINK showing values for 1 sec every 3 sec
  - Fill the grid while values keep appearing/disappearing
  - AI-proof: screenshot captures only partial info, requires mental model

Game 16: Live Route Builder (replaces Path Optimisation)[TO BUILD]
  - Nodes appear one-by-one every 2 seconds
  - Must connect each new node to existing path in real-time
  - Cannot wait to see all nodes first
  - AI-proof: no complete picture exists to screenshot

STAGE 5 - MASTERY SYNTHESIS
-----------------------------
Game 17: Memory Groups (replaces Connections)           [TO BUILD]
  - 9 words shown for 5 seconds -> ALL DISAPPEAR
  - From memory, select which 3 words belonged together
  - AI-proof: words vanish, AI never sees them

Game 18: Reflex Endurance (replaces MCQ Rapid Fire)     [TO BUILD]
  - Colored circles appear at random positions
  - Tap ONLY green ones. Speed increases every 30 seconds.
  - 2 wrong taps = game over. Lasts up to 5 minutes.
  - AI-proof: pure reflex, nothing to look up

Game 19: Pattern Survival (replaces MCQ Rapid Fire Hard)[TO BUILD]
  - Like Simon Says but with 6 colors on 3x3 grid
  - Each round adds +1 to sequence
  - 1 mistake = game over. How far can you go in 5 minutes?
  - AI-proof: real-time growing sequence, memory-based

Game 20: Speed Type Answer                              [TO BUILD]
  - Question flashes for 3 seconds -> disappears
  - User must TYPE the answer (no options) within 5 seconds
  - AI-proof: question disappears, no time to screenshot + query AI

  micro-screens/          (instruction screen, result screen - shared UI)

  src/
    game/
      game.controller.ts    (REST endpoints)
      game.service.ts       (business logic)
      game.gateway.ts       (WebSocket for real-time)
      utils/
        generator.ts      (PRNG-based content generation)

    restate/                  (durable execution workflows)
    common/                   (guards, decorators, filters)

4. SCORING SYSTEM (Per Game)
================================================================================

All scoring is SERVER-SIDE. Client shows estimated score; server calculates final.

SLIDING PUZZLE:
  Completed Round: 100 pts
  Incomplete Round: 0 pts
  Time Bonus: floor(10 x remaining_time / total_time)
  Final: Sum of round scores + time bonus

ARROWS:
  Completed Round: 50 pts
  Partially Cleared: floor(5 x arrows_removed / total_arrows)
  Empty Round: 0 pts
  Time Bonus: floor(10 x remaining_time / total_time)
  Final: Sum of round scores + time bonus

SEQUENCE RECALL:
  Score Per Click: correct_clicks x scorePerClick (default 10)
  Time Bonus: floor(seconds_left x bonusTimeRatio) - only if no wrong moves
  Final: successful_moves x scorePerClick + time_bonus

MEMORY CARD MATCHING:
  Completed Level: 100 pts
  Incomplete Level: 0 pts
  Time Bonus: remaining whole seconds (only if all levels completed)
  Final: completed_levels x 100 + time_bonus

INFINITY LOOP:
  Solved Board: 10 + remaining_seconds_on_that_board
  Unsolved Board: 0 pts
  Final: Sum of all board scores (time baked in)

BLOCK FILL:
  Easy Board: 20 pts | Medium: 30 pts | Hard: 40 pts
  Demo Board: 0 pts
  End Bonus: remaining seconds in overall timer
  Final: Sum of board scores + remaining_seconds_bonus

MAZE NAVIGATION:
  Optimal path: 100 pts
  Inefficient path: max(10, floor(100 x shortest_path / moves_taken))
  Incomplete: 0 pts
  Time Bonus: floor(5 x remaining_time / total_time)
  Final: Sum of round scores + time bonus

LOGIC REFLECTOR:
  Beam alignment score + time efficiency bonus

--- NEW GAMES (Scoring to implement) ---

FLASH SPOT:
  Correct tap: 20 pts per change identified
  Wrong tap: -10 pts
  Time Bonus: floor(10 x remaining_time / total_time)
  Final: Sum of correct taps - penalties + time bonus

OBJECT PLACEMENT MEMORY:
  Correct placement: 100 / total_objects pts per correct position
  Time Bonus: floor(5 x remaining_time / total_time)
  Final: (correct_placements / total_placements) x 100 + time_bonus

COLOUR SORTING:
  Completed: 100 pts per sorted tube
  Moves Efficiency: bonus for fewer moves
  Time Bonus: floor(10 x remaining_time / total_time)
  Final: sorted_tubes x 100 + efficiency_bonus + time_bonus

RAPID CATEGORY SORT:
  Correct sort: 10 pts per item
  Wrong sort: -5 pts
  Time Bonus: None (speed IS the score)
  Final: correct x 10 - wrong x 5

WORD UNSCRAMBLE RACE:
  Solved word: 15 pts
  Time Bonus: remaining seconds per word solved early
  Final: words_solved x 15 + sum(time_bonuses)

TRUE OR FALSE BLITZ:
  Correct: 10 pts per correct answer
  Wrong: -5 pts
  Streak bonus: 3+ correct in a row = +5 per streak item
  Final: correct x 10 - wrong x 5 + streak_bonus

NUMBER GRID SPRINT:
  Correct cell: 10 pts
  Time Bonus: remaining seconds x 2
  Final: correct_cells x 10 + time_bonus

LIVE ROUTE BUILDER:
  Path efficiency: (optimal_length / actual_length) x 100
  Nodes connected: 10 pts per node
  Final: path_score + nodes x 10

MEMORY GROUPS:
  Correct group: 50 pts per group
  Partial group: 10 pts per correct word in wrong group
  Time Bonus: remaining seconds x 3
  Final: group_scores + time_bonus

REFLEX ENDURANCE:
  Score: 1 pt per correct tap
  Speed bonus: faster taps = more points
  Final: total_correct_taps + speed_multiplier

PATTERN SURVIVAL:
  Score: round_reached x 20
  Perfect round bonus: +10 per round with no hesitation
  Final: rounds x 20 + perfect_bonuses

SPEED TYPE ANSWER:
  Correct answer: 20 pts
  Speed bonus: faster typing = bonus (max 10 extra)
  Final: correct_answers x 20 + speed_bonuses

5. GAME SESSION FLOW (API Contracts)
================================================================================

This is the communication between the Mobile App, Backend, and Game Frontend.

KEY FORMAT: contestId::playerId
  Example: eae5816f-f3d0-461e-8030-c10ee810c170::104

--- 5.1 START GAME ---
POST /UserContest/{key}/startGame
Headers: Authorization: Bearer <token>, Content-Type: application/json
Body: {
  "providerId": "<game-provider-uuid>",
  "stageId": "<stage-uuid>"
}
Response: {
  "gameSessionId": "<uuid>",
  "gameUrl": "https://game.zubaco.com/maze-navigation"
}

--- 5.2 OPEN GAME IN WEBVIEW ---
App opens gameUrl in WebView and INJECTS:
  - token (JWT)
  - gameSessionId
  - key (contestId::playerId)
Game frontend reads these to authenticate and identify the session.

--- 5.3 SUBMIT GAME RESULT ---
POST /UserContest/{key}/submitGameResult
Headers: Authorization: Bearer <token>, Content-Type: application/json
Body: {
  "gameSessionId": "<uuid>",
  "outcome": "completed",
  "score": 100
}
Response: { "success": true }

--- 5.4 GET GAME SESSION STATE ---
POST /UserContest/{key}/getGameSessionState
Body: { "gameSessionId": "<uuid>" }
Response: {
  "gameSessionId": "<uuid>",
  "gameStartedAt": "2026-05-22T09:20:52.821Z"
}

--- 5.5 COMPLETE FLOW ---
Mobile App -> startGame API -> Backend returns gameSessionId + gameUrl
Mobile App -> Opens WebView -> Injects token/gameSessionId/key
Game Frontend -> Player plays -> Submits result via API
Backend -> Validates score server-side -> Stores result -> Success

IMPORTANT RULES:
  - gameSessionId is unique per session
  - key uniquely identifies contest-player combination
  - Game Provider must use same gameSessionId received during init
  - submitGameResult only called AFTER gameplay completion
  - All injected values must be securely handled in WebView

6. FREE PLAY MODE (Levels System)
================================================================================

Purpose: Daily engagement, skill building, retention.
Available: Always (no time windows).

STRUCTURE:
  - Each game has 10 fixed levels + Endless Mode after Level 10
  - User starts at Level 1, must complete to unlock Level 2, etc.
  - Can replay any unlocked level
  - Star rating per level (1-3 stars based on score)

LEVEL CONFIG (What changes per level):
  Level 1:  Easiest params (small grid, long time, slow speed)
  Level 5:  Medium params
  Level 10: Hardest fixed params
  Level 11+: Endless mode - difficulty scales by formula

UNIVERSAL SCALING PARAMETERS:
  - Grid size increases
  - Time window decreases
  - Display/flash duration shortens
  - Number of elements increases
  - Distractors/decoys added
  - Speed of animation increases

CONTENT IS ALWAYS RANDOM:
  - Level config (difficulty) stays the same
  - Actual puzzle content is DIFFERENT every play (PRNG seed)
  - Player can NEVER memorize the answer
  - Example: Maze Level 6 = always 15x15 grid, but different maze layout each time

ENDLESS MODE (After Level 10):
  - No cap on difficulty
  - Config scales by formula: param = base + (level - 10) x increment
  - Leaderboard: "How far can you get?"
  - Example: Maze Level 20 = 20x20 grid, Level 50 = 50x50 grid

MONETIZATION HOOKS:
  - Extra lives (free play gives 3 lives/day, buy more)
  - Skip level (pay to unlock next without completing)
  - Hints (reveal 1 tile, show 1 pair, etc.)
  - Remove ads
  - Cosmetics (themes, avatar items)

IMPLEMENTATION NEEDED:
  - Level progression system (unlock tracking per user per game)
  - Star rating calculation per level
  - Level config database (10 configs per game x 20 games = 200 configs)
  - Endless mode formula per game
  - XP/progression rewards for completing levels
  - Daily limit system (energy/lives)
  - UI: Level selection screen, progress map
  - Backend: User progress persistence (which levels completed, stars)

7. TOURNAMENT MODE (Elimination System)
================================================================================

Purpose: High-stakes seasonal competition.
Available: During season windows only.

STRUCTURE:
  Season -> 5 Stages -> 4 Games per Stage -> 50% eliminated each stage

  Stage 1: 100% of players play 4 games -> bottom 50% eliminated
  Stage 2: Top 50% play 4 games -> bottom 50% eliminated (25% remain)
  Stage 3: Top 25% play 4 games -> bottom 50% eliminated (12.5% remain)
  Stage 4: Top 12.5% play 4 games -> bottom 50% eliminated (6.25% remain)
  Stage 5: Top 6.25% play 4 games -> top performers win

DIFFICULTY PER STAGE:
  - Admin configures which "level config" each game uses per stage
  - Stage 1 = Level 3 difficulty
  - Stage 3 = Level 5-6 difficulty
  - Stage 5 = Level 8-9 difficulty
  - All players in same stage get SAME difficulty config (fair)
  - Content is still randomized per player (PRNG seed unique per session)

SCORING & RANKING:
  - Each game produces a score (server-validated)
  - Stage score = sum of 4 game scores
  - Players ranked by stage score
  - Bottom 50% marked as eliminated
  - Ties broken by: total time taken (faster = better)

IMPLEMENTATION NEEDED:
  - Season CRUD (create season, set dates, assign games to stages)
  - Stage management (open/close stages, trigger elimination)
  - Game-to-stage assignment (which 4 games in each stage)
  - Difficulty config assignment (admin picks level config per stage-game)
  - Elimination engine (rank players, mark bottom 50%)
  - Tiebreaker logic (time-based)
  - Stage unlock logic (only proceed if not eliminated)
  - Results display (your rank, who got eliminated, prize info)
  - Prize distribution system
  - Re-engagement for eliminated players (spectator mode, next season)

8. SEASON & COHORT MANAGEMENT
================================================================================

SEASONS:
  - Each season has: start date, end date, prize pool, stage schedule
  - Stages open on specific dates (e.g., Stage 1: Week 1, Stage 2: Week 2...)
  - Players must complete all 4 games in a stage before deadline
  - One attempt per game per stage (one shot)

COHORTS (BUCKETING):
  - If too many players (100K+), split into cohorts
  - Each cohort runs the same stages independently
  - Winners from each cohort compete in grand finale OR
  - Top N from each cohort proceed to next stage
  - Ensures server load is manageable
  - Cohort assignment: random on registration or by region

IMPLEMENTATION NEEDED:
  - Season model (DB schema: seasons table with dates, status, prize_pool)
  - Stage model (stages table: season_id, stage_number, open_date, close_date)
  - Cohort model (cohorts table: season_id, cohort_number, player_ids)
  - Player-Season registration (user signs up for a season)
  - Cohort assignment algorithm (random/regional)
  - Stage scheduling system (auto-open/close stages)
  - Cross-cohort finale system
  - Notification system (stage opening, elimination, results)
