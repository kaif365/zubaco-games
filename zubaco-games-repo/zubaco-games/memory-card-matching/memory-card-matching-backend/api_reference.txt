MEMORY CARD MATCHING GAMEPLAY API REFERENCE
==========================================

Base URL
--------
- Local example: http://localhost:3002
- API prefix: /v1
- Swagger docs: /api


Authentication
--------------
1. All gameplay APIs require a USER bearer token.
2. The frontend should always send:
   Authorization: Bearer <token>
3. The backend verifies the token, resolves the user's Redis-backed session/profile,
   and derives the user's stageId internally.
4. The frontend should not send stageId in gameplay requests.


Gameplay Overview
-----------------
1. Frontend authenticates with a USER token.
2. Backend resolves the user's stageId from Redis-backed auth context.
3. Backend loads the active stage config for that stageId.
4. Backend resolves the playable levels from the configured level pool.
5. Backend builds the actual board, duplicates items into pairs, shuffles cards,
   and sends the shuffled cards to the frontend.
6. Restate manages the in-flight game session.


Common Notes
------------
1. All gameplay responses use the current shuffled card order generated by the backend.
2. `pairId` is the matching key the frontend should use to detect pairs locally.
3. `matchedPairIds` in save-progress/current-session responses are pair ids, not card ids.
4. Soft-deleted levels and stage configs are excluded from gameplay.
5. If Restate state is lost, the active session is considered lost.


Card Payload Shape
------------------
Every playable level returns cards like:
{
  "id": "pair-0-a",
  "pairId": "pair-0",
  "contentType": "image",
  "content": "",
  "imageUrl": "https://cdn.example.com/uploads/cat.png"
}


Playable Level Shape
--------------------
{
  "id": "level-1",
  "levelIndex": 0,
  "gridRows": 4,
  "gridColumns": 4,
  "cardContentType": "image",
  "previewDurationSeconds": 5,
  "mismatchDisplayDurationSeconds": 1,
  "cards": [ ... ]
}


========================================
GAMEPLAY APIS
========================================

GET /v1/game/session/game-configs
---------------------------------
Purpose:
- Fetch the compact gameplay config summary for the authenticated user's stage.

Auth:
- User bearer token required.

Query params:
- stageId: string, required

Response 200:
{
  "success": true,
  "statusCode": 200,
  "message": "OK",
  "data": {
    "gameTimeLimitSeconds": 180,
    "totalLevels": 3,
    "enableDemo": true
  }
}

Response shape notes:
1. This endpoint now follows the same wrapped success-response style as
   `block-fill` and `arrows`.
2. Unlike `block-fill`, the `data` object does not return the full stage-config object.
3. It currently returns only:
   - `gameTimeLimitSeconds`
   - `totalLevels`
   - `enableDemo`
4. `enableDemo` is currently hardcoded to `true`.
5. `totalLevels` is the number of playable levels resolved from the active stage config.

Common error:
- ACTIVE_STAGE_CONFIG_NOT_FOUND


POST /v1/game/session/start
---------------------------
Purpose:
- Start a new game session for the authenticated user and return the first level.

Auth:
- User bearer token required.

Request body:
{
  "stageId": "d13e13c5-cd28-4ccf-82ea-c5b71a8edd11"
}

Response 200:
{
  "success": true,
  "statusCode": 200,
  "message": "OK",
  "data": {
    "sessionId": "sess-abc123",
    "firstLevel": {
      "id": "level-1",
      "levelIndex": 0,
      "gridRows": 4,
      "gridColumns": 4,
      "cardContentType": "image",
      "previewDurationSeconds": 5,
      "mismatchDisplayDurationSeconds": 1,
      "cards": [
        {
          "id": "pair-0-a",
          "pairId": "pair-0",
          "contentType": "image",
          "content": "",
          "imageUrl": "https://cdn.example.com/uploads/cat.png"
        },
        {
          "id": "pair-2-b",
          "pairId": "pair-2",
          "contentType": "image",
          "content": "",
          "imageUrl": "https://cdn.example.com/uploads/dog.png"
        }
      ]
    }
  }
}

Notes:
1. The backend sends the board already shuffled.
2. The frontend should render the `cards` array in the received order.

-------------------------------
Purpose:
- Fetch the next level for the current in-flight game session.

Auth:
- User bearer token required.

Query params:
- sessionId: string, required

POST /v1/game/session/save-progress
-----------------------------------
Purpose:
- Save matched pair progress for the current session.

Auth:
- User bearer token required.

Request body:
{
  "sessionId": "sess-abc123",
  "levelIndex": 0,
  "matchedPairIds": ["pair-0", "pair-3", "pair-5"],
  "timeRemaining": 97
}

Response 200:
{
  "saved": true
}

Notes:
1. `matchedPairIds` must be pair ids already found by the frontend.
2. `levelIndex` must be valid for the session.
3. Backend uses this for resume support and move tracking.
4. Backend currently validates session ownership and level progression,
   not every card flip individually.

Common errors:
- NO_SESSION
- SESSION_NOT_FOUND
- INVALID_LEVEL_TRANSITION

POST /v1/game/session/game-end
------------------------------
Purpose:
- End the current game session and calculate the final result.

Auth:
- User bearer token required.

Request body:
{
  "sessionId": "sess-abc123"
}

Response 200:
{
  "success": true,
  "statusCode": 200,
  "message": "OK",
  "data": {
    "finalScore": 418,
    "rank": 82
  }
}

Notes:
1. Backend finalizes the session snapshot in DB at game end.
2. Move history collected during play is persisted at game end.
3. Result and score are derived by the backend from the active session state.


========================================
FRONTEND IMPLEMENTATION NOTES
========================================

1. The frontend receives already shuffled cards from the backend.
2. The frontend should render cards exactly in the order received in `cards[]`.
3. The frontend can determine local matches by comparing `pairId`.
4. When a match is found, send the accumulated matched `pairId`s through
   `POST /v1/game/session/save-progress`.
5. The frontend should keep and reuse the returned `sessionId` for:
   - next-level
   - save-progress
   - game-end
6. To resume an interrupted active session, call:
   `GET /v1/game/session/current`
7. Public manual test client for local gameplay testing:
   /public/game-client.html
